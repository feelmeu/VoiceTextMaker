# 0002 — 选择 Gradio 作为 GUI 框架

- 状态：Accepted
- 日期：2026-06-24

## 背景

VoiceTextMaker 需要一个图形界面，支持文件选择、参数调节、进度显示、视频预览、历史记录。需要在多个 GUI 框架中做出选择。

## 决策

我们决定采用 **Gradio** 作为 GUI 框架。

## 理由

- **AI 生态首选**：Gradio 是 AI 应用的标准 UI 框架，与 PyTorch 生态无缝集成
- **开发效率高**：用 Python 写 UI，无需前端知识，组件丰富（视频播放器、音频播放器、滑块等）
- **内置视频/音频组件**：原生支持视频预览、音频播放，无需额外集成
- **MuseTalk 官方使用**：MuseTalk 本身就用 Gradio 做 demo，集成无障碍
- **跨平台**：Web 界面，Windows/Mac/Linux 通用
- **打包友好**：可以作为本地 Web 服务运行，打包比原生 GUI 简单

## 备选方案与放弃原因

### PyQt5 / PySide

功能强大，原生桌面体验。放弃原因：
- 开发效率低，需要写大量 UI 代码
- 与 AI 模型集成复杂（线程管理、内存共享）
- 打包后体积大，依赖复杂

### Streamlit

类似 Gradio 的 Web UI 框架。放弃原因：
- 视频/音频组件不如 Gradio 丰富
- 对实时进度更新的支持不如 Gradio
- 不如 Gradio 在 AI 领域普及

### Electron + Web 前端

最灵活的方案。放弃原因：
- 开发成本过高，需要前端 + 后端双端开发
- 与 Python AI 模型集成需要额外桥接层
- 打包体积最大

## 后果

### 好处

- 开发速度快，几行代码就能搭建完整 UI
- 内置视频/音频播放器，无需额外开发
- 社区活跃，遇到问题容易找到解决方案

### 风险/代价

- UI 定制性不如原生框架
- 依赖浏览器运行
- 高级交互（拖拽、右键菜单等）支持有限

## 不做的事

- 不做自定义前端（用 Gradio 内置组件即可）
- 不做实时流式预览（首版为生成后预览）
